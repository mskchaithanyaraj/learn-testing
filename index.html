Chapter 1: Software Testing Fundamentals

What is Software?

Software is a set of instructions (programs), data, and documentation that tells a computer how to perform specific tasks. Examples: banking apps, e-commerce websites, mobile applications.

What is Software Testing?

Software Testing is the process of evaluating a software application to detect differences between the expected output and the actual output. It ensures the product is bug-free, meets requirements, and satisfies user needs.

Key Definition: Testing is not just finding bugs — it's about building confidence that the software works correctly under various conditions.

Why is Testing Needed?

  • Quality Assurance: Ensures the product meets standards and specifications
  • Cost Savings: Finding bugs early is 10-100x cheaper than fixing in production
  • Security: Identifies vulnerabilities before hackers exploit them
  • User Satisfaction: Delivers a reliable, user-friendly product
  • Compliance: Meets regulatory and industry standards
  • Reputation: Prevents costly failures that damage brand trust

Testing Techniques (Black-Box)

1. Equivalence Partitioning (EP)

Divides input data into equivalent groups/partitions where all values in a partition are expected to behave the same. You test ONE value from each partition.

Age field accepts 18-60:
• Valid partition: 18-60 (test with 30)
• Invalid partition 1: <18 (test with 10)
• Invalid partition 2: >60 (test with 75)
→ Only 3 test cases instead of testing every number!

2. Boundary Value Analysis (BVA)

Tests at the edges/boundaries of partitions because bugs are most likely at boundaries.

Age field accepts 18-60:
• Boundary values: 17, 18, 19, 59, 60, 61
• Test: just below min, min, just above min, just below max, max, just above max

3. Decision Table Testing

Used when there are multiple conditions that lead to different outcomes. Creates a table of all condition combinations.

Login: Username correct? Password correct?
• T + T = Login success
• T + F = "Invalid password"
• F + T = "Invalid username"
• F + F = "Invalid credentials"

Q: Design test cases for a login page using EP & BVA.
A: EP: Valid username + valid password (valid partition), invalid username (invalid partition), empty fields (invalid partition), special characters (invalid partition).
BVA: If username is 6-20 chars: test with 5 chars (invalid), 6 chars (valid boundary), 7 chars (valid), 19 chars (valid), 20 chars (valid boundary), 21 chars (invalid).

Chapter 3: Core Testing Types (Interview Focus)

Smoke Testing

Definition: A quick, shallow, wide test to check if the critical/core functionalities of a build work. Also called "Build Verification Testing".

Purpose: To decide whether the build is stable enough for further detailed testing.
When: Performed on EVERY new build before any other testing begins.
Who: QA team (sometimes developers).
Duration: Quick (30 min - 2 hours).
E-commerce smoke test:
✅ App launches successfully
✅ Login works
✅ Homepage loads
✅ Search returns results
✅ Add to cart works
✅ Payment page loads
Q: What happens if smoke testing fails?
A: The build is REJECTED and sent back to the development team for fixing. No further testing is performed on that build. The team waits for a new, fixed build. This saves time by not testing an unstable build.

Sanity Testing

Definition: A narrow, deep test focused on specific functionality or bug fix to verify it works correctly. It checks whether a particular section is still working after a minor change.

Purpose: To verify that a specific bug fix or feature change works without further issues.
When: After receiving a build with minor fixes or changes.
Scope: Narrow and focused (only the changed area).
Duration: Quick, targeted.
Bug was fixed: "Discount not applying on checkout"
Sanity test: Only test the discount functionality on checkout — not the entire app.

Regression Testing

Definition: Re-testing the existing functionality to ensure that new code changes, bug fixes, or enhancements have NOT broken previously working features.

When to perform:
• After bug fixes
• After new feature additions
• After code refactoring
• After environment changes
• Before every release

Exploratory Testing

Simultaneous learning, test design, and test execution without predefined test cases. Tester uses domain knowledge, intuition, and creativity to explore the application.

  • No scripts — tester explores freely
  • Good for finding unexpected bugs
  • Requires experienced testers
  • Used when requirements are incomplete or time is limited

Ad-hoc Testing

Informal testing with no planning, no documentation, no structure. Random testing to break the application.

Exploratory testing is structured exploration; Ad-hoc is completely random with no learning objective.

Chapter 4: Integration & System Testing

What is Integration Testing?

Testing the interaction between two or more modules/components that have been individually unit tested. It verifies that modules work correctly together — data flows properly, interfaces are compatible, and combined behavior is as expected.

Why needed?
• Modules developed by different teams may have interface mismatches
• Data format differences between modules
• Timing/synchronization issues
• Error handling between modules may fail

Types of Integration Testing

1. Top-Down Integration

Testing starts from the top-level modules and moves downward. Lower modules that aren't ready are replaced with stubs (dummy modules that simulate lower-level behavior).

Main Module (tested first)
↓
Sub-Module A (stub) → Sub-Module B (stub)
↓
Gradually replace stubs with real modules

2. Bottom-Up Integration

Testing starts from the lowest-level modules and moves upward. Upper modules that aren't ready are replaced with drivers (dummy modules that call lower modules).

Driver (simulates main module)
↓
Low-level Module A + Module B (tested first)
↓
Gradually replace drivers with real modules

3. Sandwich (Hybrid) Integration

Combines both top-down and bottom-up approaches. Top modules tested with stubs, bottom modules tested with drivers, then they meet in the middle.

4. Big Bang Integration

All modules integrated and tested at once. Risky — hard to isolate failures. Not recommended for large systems.

ApproachProsCons
Top-DownEarly prototype, tests critical paths firstStubs needed, lower modules tested late
Bottom-UpNo stubs needed, tests foundation firstDrivers needed, UI tested late
SandwichParallel testing, best of bothComplex to manage
Big BangSimple, no stubs/driversHard to isolate bugs, late testing

System Testing

Testing the complete, integrated application as a whole to verify it meets specified requirements. It's end-to-end testing of the entire system.

  • Tests functional AND non-functional requirements
  • Performed in an environment similar to production
  • Done by QA team (independent of developers)
  • Includes: Functional, Performance, Security, Usability testing
E-commerce System Test: User registers → Logs in → Searches product → Adds to cart → Enters address → Makes payment → Receives confirmation email → Checks order status

Interface/API Testing

Testing the communication between systems via APIs (Application Programming Interfaces). Verifies request/response, data format, error codes, authentication.

  • Tool: Postman, REST Assured
  • Tests: Status codes (200, 404, 500), response body, headers, authentication
  • Methods: GET, POST, PUT, DELETE
Q: When does integration testing start?
A: Integration testing starts after unit testing is complete for at least two modules that need to interact. In Agile, it typically begins in Sprint 2 when multiple features from Sprint 1 and Sprint 2 need to work together.

Chapter 6: Defect Lifecycle & Jira Workflow

Defect/Bug Lifecycle

New → Assigned → Open → Fixed → Retest → Verified → Closed

Alternative paths:
New → Rejected (not a bug)
New → Deferred (fix later)
New → Duplicate (already reported)
Retest → Reopened (not fixed) → Assigned again

States Explained:

  • New: Bug reported by tester, not yet reviewed
  • Assigned: Lead assigns to a developer
  • Open: Developer starts working on it
  • Fixed: Developer has fixed and deployed to test environment
  • Retest: QA verifies the fix
  • Verified: QA confirms fix works correctly
  • Closed: Bug is resolved and signed off
  • Reopened: Fix didn't work, sent back to developer
  • Deferred: Valid bug but will fix in future release
  • Rejected: Not a bug (works as designed)

Jira Workflow for Agile Teams

Backlog → To Do → In Progress → In QA → Done

Jira Concepts:

  • Epic: Large body of work (e.g., "User Management")
  • Story: User-facing feature (e.g., "As a user, I want to reset password")
  • Task: Technical work (e.g., "Set up test environment")
  • Bug: Defect found during testing
  • Sub-task: Breakdown of story/task

How QA Uses Jira:

  1. Pick story from "To Do" → move to "In Progress" (dev)
  2. Dev completes → moves to "In QA"
  3. QA tests → if pass, moves to "Done"
  4. QA tests → if fail, logs bug and links to story, moves back
  5. Bug linked to story: "Bug BUG-123 is caused by STORY-456"
Q: What proof is required before closing a defect?
A: Before closing: 1) Test case executed and passed, 2) Screenshots/video of working functionality, 3) Tested on required browsers/devices, 4) No related side effects found, 5) Regression tests passed, 6) Sign-off from test lead. All evidence attached to the Jira ticket.

Chapter 7: Agile Basics

What is Agile?

Agile is an iterative and incremental approach to software development that delivers working software in short cycles (sprints), embraces changing requirements, and promotes collaboration between cross-functional teams.

Agile Manifesto Values:
• Individuals & interactions OVER processes & tools
• Working software OVER comprehensive documentation
• Customer collaboration OVER contract negotiation
• Responding to change OVER following a plan

Scrum Framework

Scrum is the most popular Agile framework. It organizes work into fixed-length iterations called Sprints (typically 2 weeks).

Scrum Roles:

QA Role in Scrum:

  • Participates in sprint planning (estimates testing effort)
  • Reviews user stories and acceptance criteria
  • Writes test cases during the sprint
  • Tests features as they're developed (shift-left)
  • Reports bugs and verifies fixes
  • Participates in demos and retrospectives
  • Maintains regression suite

Sprint Cycle

Product Backlog → Sprint Planning → Sprint Backlog → Daily Standups → Development + Testing → Sprint Review → Sprint Retrospective → Potentially Shippable Product

Q: What are the Agile ceremonies and what happens in each?
A: There are 4 ceremonies: 1) Sprint Planning — team decides what to work on, 2) Daily Standup — 15-min sync on progress and blockers, 3) Sprint Review — demo completed work to stakeholders, 4) Sprint Retrospective — team reflects on what went well and what to improve. These ceremonies ensure transparency, inspection, and adaptation.

Chapter 9: Sprint Retrospective (Deep Dive)

Purpose

The retrospective is about continuous improvement. The team reflects on the sprint process (not the product) and identifies actionable improvements for the next sprint.

Three Key Questions

1. What went well? (Keep doing)
2. What didn't go well? (Stop or change)
3. What can we improve? (Action items for next sprint)

Examples of Retrospective Insights

What Went Well ✅What Didn't Go Well ❌Action Items 🎯
Good collaboration between dev & QABuilds deployed late, less testing timeDev to deploy builds by Wednesday
All sprint stories completedToo many bugs found in QAAdd code review before QA handoff
Automation saved regression timeRequirements were unclearPO to clarify AC before sprint planning
Daily standups were focusedEnvironment was unstableDevOps to set up dedicated QA environment

Who Participates?

  • ✅ Scrum Master (facilitates)
  • ✅ Developers
  • ✅ QA Engineers
  • ✅ Product Owner (optional but recommended)
  • ❌ Stakeholders / Management (NOT invited — safe space for team)
Q: Give an example of an improvement from retrospective.
A: In one sprint, we noticed QA was getting builds very late (Thursday) leaving only 1 day to test. In the retrospective, we agreed that developers would deploy testable features by Wednesday, giving QA 2 days. We also introduced a "dev-done" checklist to reduce bugs reaching QA. This improved our sprint completion rate from 70% to 90%.

Chapter 10: User Stories & Estimation

User Story Format

As a [type of user],
I want [some goal/action],
So that [some reason/benefit].
As a registered customer,
I want to reset my password via email,
So that I can regain access to my account if I forget my password.

INVEST Principle (Good User Story Criteria)

LetterMeaningExplanation
IIndependentCan be developed without depending on other stories
NNegotiableDetails can be discussed and refined
VValuableDelivers value to the user/business
EEstimableTeam can estimate the effort needed
SSmallCan be completed in one sprint
TTestableHas clear acceptance criteria to verify

Acceptance Criteria

Conditions that must be met for the story to be considered "done." Written in Given-When-Then format:

Given I am on the login page,
When I enter valid username and password and click Login,
Then I should be redirected to the dashboard and see a welcome message.

Story Points

A relative measure of effort required to complete a user story. NOT hours or days — it combines complexity, effort, and risk.

Fibonacci Sequence for Estimation

Teams use Fibonacci numbers: 1, 2, 3, 5, 8, 13, 21

  • 1 point: Very simple (change button color)
  • 2 points: Simple (add a validation message)
  • 3 points: Medium (create a form with validations)
  • 5 points: Complex (build search with filters)
  • 8 points: Very complex (payment integration)
  • 13 points: Extremely complex (consider breaking down)
  • 21 points: Too large — MUST be broken into smaller stories
Why Fibonacci? The gaps increase because uncertainty grows with complexity. The difference between 5 and 8 is more meaningful than between 5 and 6.

Planning Poker

A collaborative estimation technique where the whole team estimates together:

  1. PO reads the user story and acceptance criteria
  2. Team discusses and asks questions
  3. Each member privately selects a card (1, 2, 3, 5, 8, 13...)
  4. All cards revealed simultaneously
  5. If estimates differ significantly → discuss reasoning
  6. Re-vote until consensus reached
Q: What is Planning Poker and why is it used?
A: Planning Poker is a consensus-based estimation technique where team members independently estimate story points using Fibonacci cards, then reveal simultaneously. It prevents anchoring bias (where someone's opinion influences others), encourages discussion about complexity, and leverages the collective wisdom of the team. It ensures everyone's perspective is considered.

Chapter 11: Story Points & Velocity

Story Points vs Time

Story PointsTime (Hours/Days)
Relative measureAbsolute measure
Considers complexity + risk + effortOnly considers duration
Team-specific (5 points means different things to different teams)Universal (8 hours = 8 hours)
Doesn't change with experienceFaster developer = fewer hours
Accounts for unknownsOften underestimated
A 5-point story might take a senior developer 4 hours but a junior developer 12 hours. The complexity is the same (5 points) — the time differs based on skill. That's why we estimate in points, not hours.

Estimation Factors

  • Complexity: How difficult is the logic? How many edge cases?
  • Risk/Uncertainty: Are requirements clear? Is it new technology?
  • Effort: How much work is involved? How many components affected?
  • Dependencies: Does it depend on other teams or external systems?

Velocity

Definition: The average number of story points a team completes per sprint.

Sprint 1: Completed 20 points
Sprint 2: Completed 25 points
Sprint 3: Completed 22 points
Average Velocity = (20 + 25 + 22) / 3 = ~22 points/sprint

How Velocity Impacts Sprint Planning:

  • If velocity is 22, the team should commit to ~22 points of work next sprint
  • Helps predict how many sprints needed to complete the backlog
  • Used for release planning: "With 100 points remaining and velocity of 22, we need ~5 sprints"
  • Velocity stabilizes after 3-4 sprints — use it for forecasting
Q: How does the Scrum Master help with estimation?
A: The Scrum Master facilitates Planning Poker, ensures discussions stay productive, helps the team understand the story, and prevents pressure from PO/management to underestimate. They do NOT estimate themselves (unless also doing development work). They coach the team on relative estimation and remind them to consider complexity, risk, and uncertainty — not just time.

Chapter 12: Story Completion & Definition of Done (DoD)

Definition of Done (DoD)

A shared checklist that defines when a user story is truly "complete." It's agreed upon by the entire team and applies to ALL stories.

Typical Definition of Done:
✅ Code written and peer-reviewed
✅ Unit tests written and passing
✅ Feature deployed to QA environment
✅ All acceptance criteria verified by QA
✅ No open critical/high bugs
✅ Regression tests passed
✅ Documentation updated (if needed)
✅ Demo-ready for sprint review

Who Marks a Story as Done?

  • Developer: Marks as "Ready for QA" after coding & unit testing
  • QA: Tests and moves to "Done" if all acceptance criteria pass
  • Product Owner: Final acceptance in Sprint Review (validates business value)
  • Scrum Master: Does NOT mark stories as done — they facilitate, not manage
Q: Can the Scrum Master mark a story as complete?
A: No. The Scrum Master is a facilitator, not a decision-maker on completeness. The Development Team (including QA) determines if the DoD is met. The Product Owner validates that the acceptance criteria satisfy the business need. The Scrum Master only ensures the process is followed correctly.

Chapter 13: Regression Strategy

When to Run Regression

  • After every bug fix (to ensure fix didn't break anything)
  • After new feature addition
  • After code refactoring or optimization
  • Before every release/deployment
  • After environment changes (server, database, OS update)
  • After merging code branches

Regression Test Suite

A maintained collection of test cases that covers critical business flows and previously found bug areas. It grows over time.

How to Build a Regression Suite:

  1. Core flows: Login, Registration, Payment, Search (always include)
  2. Bug-prone areas: Modules where bugs were frequently found
  3. High-business-value features: Revenue-generating features
  4. Integration points: Where modules connect
  5. Recently changed areas: Code that was recently modified

Manual vs Automation Regression

AspectManual RegressionAutomated Regression
SpeedSlow (hours/days)Fast (minutes/hours)
Cost per runHigh (human effort each time)Low (once built, runs free)
AccuracyHuman errors possibleConsistent every time
Best whenSuite is small, UI changes frequentlySuite is large, stable features
CI/CDCannot integrateRuns automatically on every build

CI/CD Integration

CI (Continuous Integration): Developers merge code frequently; automated tests run on every merge.

CD (Continuous Delivery/Deployment): Code is automatically deployed after tests pass.

Developer pushes code → CI server builds → Automated regression runs → If pass: deploy to QA → If fail: notify team
Q: What are the challenges in regression testing?
A: 1) Time-consuming — suite grows with every sprint, 2) Maintaining test cases as UI/features change, 3) Deciding what to include (can't test everything), 4) Flaky tests giving false failures, 5) Environment dependencies. Solutions: Prioritize critical paths, automate stable tests, use risk-based testing approach, maintain test data properly.

Chapter 14: Basics of Automation Testing

Why Automation is Needed

  • Manual regression takes too long as the app grows
  • Repetitive tests are boring and error-prone for humans
  • Faster feedback in CI/CD pipelines
  • Run tests 24/7 without human intervention
  • Cross-browser/device testing at scale
  • Consistent and reliable results

Test Pyramid

      / UI Tests \    ← Few, slow, expensive
    / Integration Tests \  ← Medium
  /    Unit Tests      \ ← Many, fast, cheap
  • Unit Tests (Base — 70%): Fast, test individual functions, run by developers
  • Integration/API Tests (Middle — 20%): Test module interactions, API responses
  • UI/E2E Tests (Top — 10%): Slow, brittle, test complete user flows
The pyramid suggests: have MANY unit tests, SOME integration tests, and FEW UI tests. Most bugs should be caught at lower levels.

Basic Automation Framework Concepts

  • Page Object Model (POM): Each page has a class with its elements and actions
  • Data-Driven Testing: Same test runs with different data sets
  • Keyword-Driven: Actions described by keywords (Click, Enter, Verify)
  • Hybrid Framework: Combination of above approaches
Q: What test cases should be automated first?
A: 1) Smoke tests — run on every build, 2) Regression tests for stable features — repetitive and time-consuming, 3) Data-driven tests — same flow with many data combinations, 4) API tests — fast ROI, less maintenance than UI tests. Start with high-value, stable, frequently-run tests. Avoid automating tests for features still changing rapidly.

Q: Explain how testing evolves in Agile sprints.
A: In Agile, testing starts from day 1 and evolves each sprint. Sprint 1 focuses on understanding the product and exploratory testing. Sprint 2 introduces integration testing as modules connect. Sprint 3 adds non-functional testing (security, performance) as the product stabilizes. Sprint 4 focuses on UAT and release readiness. The QA role evolves from Collaborator → Builder → Quality Advocate → Release Gatekeeper. Testing is continuous, not a phase — it's embedded in every sprint activity.
